FFI

Jackey Zhou2025-05-082025-05-08

外部函数接口 (Foreign Function Interface, FFI) 是 Rust 语言中一个非常重要的特性，它允许你的 Rust 代码与其他编程语言（主要是那些遵循 C 语言应用程序二进制接口 ABI 的语言）进行交互。这意味着你可以：

  1. 从 Rust 代码中调用其他语言编写的函数（例如，复用一个用 C 语言编写的成熟库）。
  2. 将 Rust 代码编译成库，供其他语言调用（例如，用 Rust 编写高性能模块，然后在 Python 或 Node.js 项目中使用）。

FFI 本质上是 unsafe 的，因为 Rust 编译器无法验证外部语言代码的内存安全保证或其行为是否符合 Rust 的规则。因此，在使用 FFI 时，你需要承担保证交互边界安全的责任。

1. Rust 调用外部代码 (主要是 C 语言函数)

当你想在 Rust 中使用一个用 C 语言（或其他兼容 C ABI 的语言）编写的库时，你需要做以下几步：

a) 声明外部函数 (extern "C" 块)

你需要告诉 Rust 编译器外部函数的签名（名称、参数类型、返回类型）。这通过 extern "C" 块来完成。”C” 表示使用 C 语言的调用约定和 ABI。

注意:

  • std::os::raw::c_char (或 libc::c_char，如果使用 libc crate) 是与 C 语言 char 类型对应的 Rust 类型。
  • C 语言中的字符串通常是空终止 (null-terminated) 的字符数组。在 Rust 中，std::ffi::CString 用于创建这种与 C 兼容的字符串，而 std::ffi::CStr 用于包装从 C 接收到的空终止字符串指针。

b) 链接到外部库

仅仅声明外部函数是不够的，你还需要告诉 Rust 编译器（以及链接器）在哪里找到这些函数的实际实现。

  1. #[link] 属性:
    你可以使用 #[link(name = "libraryname")] 属性来指定要链接的库名。

    Cargo 默认会尝试动态链接。如果需要静态链接，可以使用 #[link(name = "libraryname", kind = "static")]。

  2. 构建脚本 (build.rs):
    对于更复杂的场景，例如需要编译 C 代码、配置链接器路径或根据不同平台选择不同库时，通常会使用构建脚本 (build.rs)。
    构建脚本是一个位于项目根目录的 Rust 文件，Cargo 会在编译你的包之前先编译并运行它。
    你可以使用像 cc crate 这样的辅助库来方便地编译 C/C++/汇编代码并链接。

c) 数据类型转换 (Marshaling)

在 Rust 和 C 之间传递数据时，需要确保类型兼容。

  • 基本数值类型:
    • Rust 的 i8, u8, i16, u16, i32, u32, i64, u64, f32, f64 通常可以直接对应 C 中的 int8_t, uint8_t, int16_t, uint16_t, int32_t, uint32_t, int64_t, uint64_t, float, double。
    • 使用 std::os::raw (例如 c_int, c_char, c_double) 或 libc crate 中的类型别名可以提高可移植性和明确性。
  • 指针:
    • Rust 的 *const T 对应 C 的 const T*。
    • Rust 的 *mut T 对应 C 的 T*。
    • Rust 的引用 (&T, &mut T) 在 FFI 边界上传递时，如果生命周期管理正确，可以转换为裸指针。
  • 字符串:
    • Rust String / &str -> C const char*:
      1. 使用 std::ffi::CString::new(rust_string) 创建一个包含空终止符的 C 兼容字符串。
      2. 使用 cstring.as_ptr() 获取 *const c_char。
      3. 注意: CString 的生命周期必须覆盖外部函数使用该指针的整个期间，否则会导致悬垂指针。
    • C const char* -> Rust &str / String:
      1. 如果 C 字符串指针是有效的且空终止的，可以使用 unsafe { std::ffi::CStr::from_ptr(c_ptr) } 创建一个 &CStr。
      2. 然后使用 c_str.to_str()? (返回 Result<&str, Utf8Error>) 或 c_str.to_string_lossy() (返回 Cow<str>) 转换为 Rust 字符串。
  • 结构体 (struct):
    • 默认情况下，Rust 编译器可能会为了优化而重排结构体字段的内存布局。C 编译器也有其自己的布局规则。
    • 为了确保 Rust 结构体与 C 结构体在内存布局上兼容，Rust 结构体需要使用 #[repr(C)] 属性。
  • 枚举 (enum):
    • C 风格的枚举（只包含判别式）可以映射到 Rust 的整数类型。
    • 如果 Rust 枚举带有数据，则没有直接的 C 等价物，需要手动转换或定义更复杂的 FFI 接口。
    • 使用 #[repr(C)] 或 #[repr(IntType)] 可以控制 Rust 枚举的内存表示。
  • 函数指针 (Callbacks):
    • 如果 C 函数期望一个回调函数作为参数，你需要在 Rust 中定义一个兼容的函数（通常也需要 extern "C" 调用约定），然后将其作为函数指针传递。

d) 内存管理

当通过 FFI 在 Rust 和 C 代码之间传递拥有内存的指针时（例如，C 函数返回一个需要释放的字符串指针，或者 Rust 将 CString 的指针传递给 C），必须明确哪一方负责分配和释放内存。

  • C 分配，Rust 释放: 如果 C 函数分配了内存，它通常应该也提供一个对应的释放函数，Rust 代码需要通过 FFI 调用这个释放函数。
  • Rust 分配，C 释放: 不推荐，因为 Rust 和 C 可能使用不同的内存分配器。最好是 Rust 分配，Rust 释放。如果必须让 C 释放，Rust 需要提供一个 extern "C" 的释放函数，该函数接收指针并正确地重建 Rust 的数据结构（如 Box 或 CString）以便 drop。
  • Rust 分配，Rust 释放，C 临时使用: 这是最安全的模式。Rust 分配数据，将指针传递给 C，C 使用完后，Rust 负责释放。

2. 将 Rust 代码暴露给外部 (主要是 C 语言)

你也可以将 Rust 函数编译成库，让 C 或其他语言调用。

a) 定义可供外部调用的 Rust 函数

  • 使用 pub extern "C" 来声明函数，使其具有 C ABI。

c) C 代码中调用 Rust 函数

你需要一个头文件来声明 Rust 函数的签名，然后在 C 代码中包含这个头文件并链接到 Rust 生成的库。

编译 C 代码时，需要链接 Rust 生成的库：
gcc main.c -L/path/to/rust_target_dir -lmy_rust_lib -o main_c_app (具体命令取决于操作系统和库类型)

d) 处理 Panic

默认情况下，如果 Rust 代码在 FFI 边界发生 panic，行为是未定义的 (通常会导致程序中止)。这是非常不希望的。
你应该在 extern "C" 函数的边界处使用 std::panic::catch_unwind 来捕获 panic，并将其转换为一个错误码或其他 C 兼容的错误表示方式返回给调用者。

3. FFI 的安全性和注意事项

  • unsafe 是核心: 所有 FFI 调用几乎都涉及 unsafe，因为编译器无法验证外部代码。
  • 数据类型和内存布局: 确保 #[repr(C)] 用于结构体，并正确映射类型。
  • 字符串处理: CString 和 CStr 是你的朋友。注意空终止符和 UTF-8 编码问题。
  • 所有权和内存管理: 明确哪一方负责分配和释放内存。避免内存泄漏和悬垂指针。
  • 错误处理: Rust 的 Result 和 panic! 需要转换为 C 兼容的错误码或约定。
  • 回调函数: 确保回调函数的生命周期和传递给它的任何数据的生命周期是正确的。传递给 C 的数据（如果回调函数会访问它）通常需要是 'static，或者由调用者保证其在回调期间有效。
  • 空指针检查: C 函数可能会返回空指针，或期望空指针作为哨兵值。Rust 代码需要妥善处理。
  • 线程安全: 如果你调用的 C 库不是线程安全的，你需要在 Rust 端确保同步访问。如果 Rust 库要被多线程的 C 程序调用，Rust 代码本身也需要是线程安全的，或者提供线程安全的接口。

4. FFI 辅助工具和 Crate

  • libc: 提供了 C 标准库中许多类型和函数的 Rust 声明 (例如 libc::c_char, libc::size_t, libc::malloc)。
  • bindgen: 一个非常有用的工具（通常作为构建脚本的一部分或命令行工具使用），它可以自动从 C/C++ 头文件生成 Rust 的 FFI 绑定。
  • cbindgen: 与 bindgen 相反，它可以从你的 Rust 库代码生成 C/C++ 兼容的头文件，方便其他语言调用你的 Rust 库。
  • safer_ffi: 旨在帮助编写更类型安全、更符合人体工程学的 FFI 绑定。
  • 特定语言的 FFI 库:
    • Python: PyO3, rust-cpython
    • Node.js/JavaScript: Neon, wasm-bindgen (用于 WebAssembly)
    • Ruby: Magnus, Rutie, rb-sys
    • Java/JVM: JNI crate
    • C#/.NET: P/Invoke (平台调用)

总结

FFI 是 Rust 与现有代码世界连接的桥梁。它使得 Rust 能够逐步被采用，并利用数十年来积累的成熟库。然而，FFI 要求开发者对两种语言的底层机制都有深入的理解，并承担起维护 FFI 边界内存安全的责任。通过 unsafe 块、仔细的数据类型转换、明确的内存管理约定以及 std::panic::catch_unwind 等工具，可以构建出相对健壮的 FFI 接口。
