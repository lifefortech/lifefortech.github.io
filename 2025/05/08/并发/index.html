并发

Jackey Zhou2025-05-082025-05-08

并发 (Concurrency) 和多线程 (Multithreading) 是现代软件开发中非常重要的主题，它们能够帮助我们构建响应更快、性能更高的应用程序。Rust 在这方面提供了强大的支持，并以其“无畏并发 (Fearless Concurrency)”而闻名，这意味着 Rust 的类型系统和所有权模型能够在编译时捕获许多常见的并发错误，如数据竞争。

让我们详细探讨 Rust 中的并发和多线程：

1. 并发 (Concurrency) vs. 并行 (Parallelism)

首先，区分这两个概念很重要：

  • 并发 (Concurrency): 指的是多个任务可以交替执行或重叠执行的能力，使得它们看起来像是在同时运行。这可以在单个处理器核心上通过时间分片实现，或者在多个核心上实现。并发更侧重于程序结构的设计，使其能够处理多个独立进行的工作单元。
  • 并行 (Parallelism): 指的是多个任务真正同时执行的能力，这通常需要多个处理器核心。并行是并发的一种实现方式，旨在通过同时执行计算来提高性能。

Rust 提供了创建并发执行代码的工具，如果底层硬件支持，这些代码可以并行运行。

2. 创建和管理线程 (std::thread)

Rust 标准库通过 std::thread 模块提供了创建和管理原生操作系统线程的功能。

a) thread::spawn()

thread::spawn() 函数用于创建一个新线程。它接收一个闭包作为参数，这个闭包包含了新线程将要执行的代码。

move 闭包:
当新线程需要使用其外部作用域（例如 main 函数作用域）中的数据时，你需要使用 move 关键字来强制闭包获取这些数据的所有权。这是因为新线程的生命周期可能比创建它的线程更长，如果只是借用，可能会导致悬垂引用。

JoinHandle<T>:
thread::spawn() 返回一个 JoinHandle<T>。T 是闭包的返回类型。JoinHandle 的 join() 方法会阻塞当前线程，直到它所代表的线程执行完毕。如果被 join 的线程 panic! 了，join() 会返回一个 Err。

b) 线程的 Panic

如果一个线程 panic! 了，它不会直接导致整个程序崩溃（除非它是主线程且未被其他线程 join）。Rust 会展开该线程的栈并清理其资源。你可以通过 std::panic::catch_unwind 来捕获子线程的 panic（但这通常用于特定的库或框架逻辑，而不是常规的错误处理）。

3. 消息传递进行线程间通信 (std::sync::mpsc)

Rust 推荐使用消息传递 (message passing) 的方式在线程间进行通信，而不是依赖共享内存。这种方式下，数据的所有权在线程间通过“通道 (channels)”进行传递，有助于避免复杂的同步问题。

std::sync::mpsc 模块提供了一个多生产者、单消费者 (multiple producer, single consumer) 的 FIFO 队列实现。

  • channel(): 函数创建一个异步通道，返回一个元组 (Sender<T>, Receiver<T>)。
    • Sender<T> (发送端): 用于发送 T 类型的消息。它可以被克隆 (clone())，从而允许多个线程向同一个接收端发送消息。
    • Receiver<T> (接收端): 用于接收消息。
  • 发送数据: sender.send(value: T) 方法发送一个值。这个操作会转移 value 的所有权。
  • 接收数据:
    • receiver.recv() -> Result<T, RecvError>: 阻塞当前线程，直到有消息可用或通道关闭。
    • receiver.try_recv() -> Result<T, TryRecvError>: 非阻塞版本，立即返回一个 Result，如果此时没有消息，会返回 Err(TryRecvError::Empty)。

当所有的 Sender 都离开作用域（被 drop）后，通道会自动关闭，此时 Receiver 端的 recv() 或迭代会结束。

sync_channel(bound: usize): 创建一个同步通道，其缓冲区大小是有限的。如果缓冲区已满，send() 操作会阻塞，直到有空间可用。

4. 共享状态并发 (Shared-State Concurrency)

虽然消息传递是推荐的方式，但有时共享内存（状态）也是必要的。Rust 提供了确保共享状态访问安全的机制，主要是通过 Mutex<T> 和 RwLock<T>，并通常与 Arc<T> 结合使用。

a) Mutex<T> (互斥锁 / Mutual Exclusion)

Mutex<T> 确保在任何时候只有一个线程可以访问 Mutex 内部的数据 T。

  • lock(): 尝试获取互斥锁。
    • 如果锁未被持有，当前线程获得锁，并返回一个 LockResult<MutexGuard<T>>。
    • 如果锁已被其他线程持有，当前线程会阻塞，直到锁被释放。
    • 如果持有锁的线程 panic! 了，Mutex 会被标记为“被污染 (poisoned)”，后续的 lock() 调用会返回 Err。你可以选择 unwrap()（如果认为污染是可接受的 bug）或处理这个 Err。
  • MutexGuard<T>: 这是一个智能指针，它实现了 Deref 和 DerefMut，允许你访问被保护的数据。当 MutexGuard<T> 离开作用域时，它会自动释放锁（通过 Drop trait）。

b) RwLock<T> (读写锁 / Read-Write Lock)

RwLock<T> 允许多个线程同时读取数据，或者只允许一个线程写入数据。

  • read(): 获取一个读锁。可以有多个读锁同时存在。返回 LockResult<RwLockReadGuard<T>>。
  • write(): 获取一个写锁。写锁是独占的，当持有写锁时，不能有其他读锁或写锁。返回 LockResult<RwLockWriteGuard<T>>。
  • 如果请求的锁与当前持有的锁冲突（例如，请求写锁时存在读锁，或请求读锁时存在写锁），则当前线程会阻塞。

c) Arc<T> (原子引用计数) 与锁的结合

要在多个线程之间共享 Mutex<T> 或 RwLock<T> 本身（因为它们管理的数据可能需要在多个线程中被访问），你需要使用 Arc<T> 来包装它们。Arc 提供了线程安全的共享所有权。

常见的模式是 Arc<Mutex<T>> 或 Arc<RwLock<T>>。

5. Send 和 Sync Trait

Rust 的并发安全性很大程度上依赖于两个特殊的**标记 trait (marker traits)**：Send 和 Sync。这些 trait 没有方法，它们只是用来标记类型是否满足某些并发相关的属性。编译器会自动为大多数类型推导它们。

  • Send:

    • 如果一个类型 T 实现了 Send，则意味着 T 类型的值的所有权可以安全地从一个线程转移到另一个线程。
    • 大多数基本类型（如 i32, bool）、String、Vec<T> (如果 T 是 Send)、Box<T> (如果 T 是 Send) 都是 Send。
    • Rc<T> 和 RefCell<T> 不是 Send，因为它们的内部机制不是线程安全的。

  • Sync:

    • 如果一个类型 T 实现了 Sync，则意味着 T 类型的不可变引用 &T 可以安全地在多个线程之间共享。
    • 换句话说，如果 T 是 Sync，那么 &T 就是 Send。
    • 大多数基本类型、String、&'static str 都是 Sync。
    • 如果一个类型的所有字段都是 Sync，那么该类型通常也是 Sync。
    • Cell<T> 和 RefCell<T> 不是 Sync，因为即使通过不可变引用访问它们，也可能导致内部状态的非同步改变。
    • Mutex<T> 是 Sync（如果 T 是 Send），即使它允许内部可变性，因为 Mutex 自身提供了同步。

编译器会利用 Send 和 Sync 来确保你不会在线程间不安全地传递或共享数据。例如，thread::spawn 要求其闭包是 Send，并且闭包捕获的任何数据也必须是 Send（或者通过 move 转移所有权）。

6. 异步编程 (async/await)

对于大量 I/O 密集型任务（如网络请求、文件读写），创建大量操作系统线程可能会消耗过多资源。异步编程提供了一种更轻量级的方式来处理并发，它允许单个线程在等待 I/O 操作完成时执行其他任务，而不是阻塞。

  • async fn: 定义一个异步函数。异步函数在被调用时不会立即执行，而是返回一个实现了 Future trait 的值。Future 代表一个未来某个时刻会产生结果的计算。
  • .await: 在 async 函数或 async 块内部使用，用于等待一个 Future 完成。当 .await 一个 Future 时，如果该 Future 尚未准备好（例如，正在等待网络响应），当前任务会被挂起，允许执行器 (executor) 运行其他准备好的任务。一旦 Future 完成，任务会从 .await 的地方继续执行。
  • 执行器 (Executor / Runtime): async/await 代码需要一个异步运行时（如 tokio, async-std, smol）来轮询 Future 并驱动它们完成。Rust 标准库本身不提供异步运行时。

异步编程是一个庞大的主题，涉及到 Future trait、Pin、任务调度等更深层次的概念。

7. 其他并发原语和库

  • std::sync::atomic: 提供原子类型（如 AtomicBool, AtomicIsize, AtomicPtr），它们支持无锁的原子操作，适用于低级并发控制。
  • std::sync::Barrier: 允许多个线程相互等待，直到所有线程都到达一个屏障点。
  • std::sync::Condvar: 条件变量，与 Mutex 一起使用，允许线程在某个条件变为真之前挂起，并在条件满足时被唤醒。
  • 外部库:
    • crossbeam: 提供了更多高级的并发数据结构（如无锁队列、epoch-based GC）和工具。
    • rayon: 提供了简单易用的数据并行库，可以将计算密集型任务自动分配到多个核心上并行处理（基于工作窃取算法）。

总结

Rust 通过其所有权系统、类型系统（特别是 Send 和 Sync trait）以及标准库提供的工具（如 thread, mpsc, Mutex, Arc），为编写安全且高效的并发代码提供了坚实的基础。

  • 对于数据隔离和避免共享状态的复杂性，优先考虑消息传递。
  • 当确实需要共享状态时，使用 Arc 来共享所有权，并使用 Mutex 或 RwLock 来同步对数据的访问。
  • 对于 I/O 密集型任务，考虑使用 async/await 和异步运行时。
