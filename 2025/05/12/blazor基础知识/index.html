1. Blazor 简介

什么是 Blazor?

Blazor 是一个使用 C# 和 .NET 构建交互式客户端 Web UI 的框架。它允许开发者使用 C# 代替 JavaScript 来编写 Web 应用程序的客户端逻辑。Blazor 应用由可重用的 Web UI 组件构成，这些组件使用 C#、HTML 和 CSS 实现。

核心优势

  • 使用 C# 进行全栈开发: 可以在客户端和服务器端共享代码和库。
  • 利用现有 .NET 生态系统: 可以使用成熟的 .NET 库和工具。
  • 可重用的 UI 组件: 构建模块化、可维护的 UI。
  • 性能: Blazor WebAssembly 直接在浏览器中运行 .NET 代码，接近原生性能；Blazor Server 通过 SignalR 连接将 UI 更新的计算工作放在服务器上。
  • 强大的工具支持: Visual Studio 和 Visual Studio Code 提供了出色的开发体验。
  • 活跃的社区: 拥有不断增长的社区和丰富的学习资源。

2. Blazor 核心概念: 组件 (Components)

Blazor 应用的核心是组件。组件是自包含的用户界面 (UI) 块，例如页面、对话框或数据输入表单。组件通常以 .razor 文件形式编写，包含 Razor 标记、C# 代码和 CSS。

Razor 语法

Razor 是一种标记语法，用于将 C# 代码嵌入到 HTML 标记中。

  • @page "/counter": 指令，将此组件设置为可通过指定路由访问的页面。
  • @currentCount: 在 HTML 中渲染 C# 变量的值。
  • @onclick="IncrementCount": 将按钮的点击事件绑定到 C# 方法 IncrementCount。
  • @code { ... }: C# 代码块，包含组件的逻辑和成员。

组件也可以有单独的 .razor.cs 代码隐藏文件。

组件生命周期

Blazor 组件有一套明确的生命周期方法，可以在组件的不同阶段执行代码：

  • SetParametersAsync(ParameterView parameters): 在组件接收到参数时调用。这是设置属性（包括从参数设置的属性）的正确位置。通常应调用 base.SetParametersAsync(parameters)。
  • OnInitializedAsync() / OnInitialized(): 在组件初始化完成并且已接收其初始参数后调用。Async 版本用于异步操作（如数据加载）。此方法每个组件实例只执行一次。
  • OnParametersSetAsync() / OnParametersSet(): 在组件接收到来自其父组件的参数并且值已分配给属性后调用。如果父组件重新渲染并导致子组件参数更改，则此方法可能会被多次调用。
  • OnAfterRenderAsync(bool firstRender) / OnAfterRender(bool firstRender): 在组件完成渲染后调用。firstRender 参数指示这是否是组件的第一次渲染。此时可以执行 JavaScript 互操作和与 DOM 元素的交互。
  • ShouldRender(): 可以重写此方法以控制组件是否应在状态更改后重新渲染。返回 true 则渲染，false 则跳过。
  • Dispose() / DisposeAsync(): 当组件从 UI 中移除时调用，用于释放资源。实现 IDisposable 或 IAsyncDisposable。

组件参数 (Parameters)

组件可以使用 [Parameter] 特性声明参数，以便从父组件接收数据。

子内容 (Child Content) 与 RenderFragment

组件可以接收和渲染由父组件提供的 UI 内容。这通过 RenderFragment 类型的参数实现，通常命名为 ChildContent。

可以定义多个 RenderFragment 参数以允许父组件提供多个 UI 片段。

级联参数 (Cascading Parameters)

级联参数允许一个组件向其所有后代组件传递值，而无需显式地将参数逐层传递。

ThemeInfo 是一个自定义类。

模板化组件 (Templated Components)

模板化组件允许组件的消费者提供 UI 模板，而不仅仅是简单的内容。这是通过 RenderFragment<T> 类型的参数实现的。

@typeparam TItem 声明了一个类型参数。Context="product" 在模板中为当前项命名。

3. Blazor 托管模型与渲染模式 (.NET 8+)

传统托管模型回顾

Blazor Server

  • 工作方式: UI 在服务器上运行，UI 更新、事件处理和 JavaScript 调用通过 SignalR 连接（实时双向通信库）在服务器和浏览器之间传输。
  • 优点:
    • 下载体积小，初始加载快。
    • 直接访问服务器资源和完整的 .NET Core API。
    • 代码保留在服务器上，更安全。
  • 缺点:
    • 需要活动的服务器连接，离线不支持。
    • 每个用户连接都会消耗服务器资源。
    • 延迟较高，因为每个交互都需要网络往返。

Blazor WebAssembly (WASM)

  • 工作方式: C# 代码和 .NET 运行时直接在浏览器中使用 WebAssembly 执行。下载完成后，应用可以在客户端离线运行（如果设计如此）。
  • 优点:
    • 真正的客户端应用，可以在断网后继续运行（如果设计如此）。
    • 利用客户端资源，减少服务器负载。
    • 可以部署为静态站点。
  • 缺点:
    • 初始下载体积较大（.NET 运行时和应用 DLL）。
    • 加载时间可能较长。
    • 浏览器沙箱限制，对硬件和系统资源的访问受限。
    • 调试和性能分析可能不如服务器端直接。

Blazor Hybrid

  • 工作方式: Blazor Hybrid 允许将 Blazor 组件嵌入到本机客户端应用程序（如 .NET MAUI, WPF, Windows Forms）中。组件在嵌入式 Web View 中本地运行，但可以访问本机功能和资源。
  • 优点:
    • 重用 Web UI 组件构建桌面和移动应用。
    • 完全访问本机设备功能。
    • 可以在现有本机应用中逐步采用 Blazor。
  • 缺点:
    • 需要特定平台的构建和部署。
    • 应用包体积可能较大。

.NET 8 中的统一模型和渲染模式

.NET 8 引入了更灵活的渲染模式，允许开发者在同一应用中为不同组件选择不同的渲染策略，甚至可以在运行时动态切换。

静态服务器端渲染 (Static SSR)

  • 工作方式: 组件在服务器上执行，并生成静态 HTML，然后发送到客户端。没有交互性，事件处理（如 @onclick）不起作用。
  • 优点:
    • 快速初始加载，对 SEO 友好。
    • 服务器资源占用低。
  • 适用场景: 展示内容为主的页面，如博客文章、产品列表。
  • 默认情况下，.NET 8 Blazor Web 应用中的组件使用静态 SSR。

流式渲染 (Streaming Rendering)

  • 工作方式: 建立在静态 SSR 之上。如果页面需要执行耗时的异步任务（如数据库查询）来完全呈现，服务器会首先发送占位符内容的初始 HTML 响应。一旦异步任务完成，更新的内容会通过同一响应流发送到客户端，并由 Blazor 的轻量级 JavaScript 自动更新到 DOM 中。
  • 优点:
    • 改善感知性能，用户更快看到内容。
    • 仍然是服务器渲染，对 SEO 友好。
  • 配置: 在组件上使用 @attribute [StreamRendering(true)]。

交互式服务器渲染 (Interactive Server)

  • 工作方式: 这与传统的 Blazor Server 模型相同。组件通过 SignalR 连接实现完全交互。
  • 优点:
    • 完整的 .NET API 访问。
    • 下载体积小。
  • 适用场景: 需要完整交互性且可以保持服务器连接的组件。
  • 配置: 在组件上使用 @rendermode InteractiveServer 或 @rendermode RenderMode.InteractiveServer。

交互式 WebAssembly 渲染 (Interactive WebAssembly)

  • 工作方式: 这与传统的 Blazor WebAssembly 模型相同。组件在客户端使用 WebAssembly 运行。
  • 优点:
    • 客户端处理，减少服务器负载。
    • 支持离线。
  • 适用场景: 需要丰富客户端交互、可能需要离线支持的组件。
  • 配置: 在组件上使用 @rendermode InteractiveWebAssembly 或 @rendermode RenderMode.InteractiveWebAssembly。需要确保项目已配置为支持 WASM。

自动渲染模式 (Interactive Auto)

  • 工作方式: 最初使用服务器端渲染 (Interactive Server) 以快速加载。同时，.NET 运行时和应用包会在后台下载到客户端。下载完成后，后续的交互会切换到 WebAssembly (Interactive WebAssembly) 执行。
  • 优点:
    • 结合了 Server 模式的快速初始加载和 WASM 模式的长期客户端性能优势。
  • 适用场景: 希望提供最佳初始加载体验和长期交互性能的组件。
  • 配置: 在组件上使用 @rendermode InteractiveAuto 或 @rendermode RenderMode.InteractiveAuto。需要确保项目已配置为同时支持 Server 和 WASM。

4. 数据绑定 (Data Binding)

数据绑定是将组件的属性或字段与 HTML 元素的值连接起来的过程。

单向绑定

从组件的 C# 成员到 HTML。

双向绑定 (@bind)

允许数据在组件的 C# 成员和 HTML 输入元素之间双向流动。

@bind 默认在 onchange 事件触发时更新。

@bind:event

可以指定何时更新绑定的值。

@bind:format

用于格式化绑定到输入字段的日期和数字。

5. 事件处理 (Event Handling)

HTML DOM 事件 (@on{event})

Blazor 使用 @on{event} 指令将 HTML 事件（如 click, input, submit）绑定到 C# 方法。

事件参数 (EventArgs)

事件处理方法可以接受特定类型的 EventArgs（如 MouseEventArgs, KeyboardEventArgs, ChangeEventArgs）以获取事件的详细信息。

异步事件处理

事件处理方法可以是异步的，返回 Task。

StateHasChanged() 通知组件其状态已更改，应重新渲染。在 Blazor Server 和 Interactive Server 模式下，通常会自动调用。在 Blazor WebAssembly 和 Interactive WASM/Auto 模式下，如果状态更改发生在 Blazor 的同步上下文之外（例如，计时器回调、非 Blazor 管理的异步操作完成时），则可能需要手动调用。

阻止默认行为与停止事件传播

  • @on{event}:preventDefault: 阻止事件的默认浏览器行为。
  • @on{event}:stopPropagation: 停止事件向上冒泡到父元素。

6. 路由与导航 (Routing and Navigation)

@page 指令

@page 指令使组件成为一个可路由的页面。

路由参数

路由可以包含参数，这些参数会传递给组件的相应属性。

路由约束

可以对路由参数应用约束，以确保它们是特定类型。

NavLink 组件

NavLink 组件用于创建导航链接。它会自动在链接的 URL 与当前 URL 匹配时添加 active CSS 类。

  • NavLinkMatch.All: 仅当整个 URL 完全匹配时激活。
  • NavLinkMatch.Prefix: 当 URL 以 href 为前缀时激活（默认）。

NavigationManager 服务

NavigationManager 服务提供了以编程方式导航和处理 URI 的方法。

查询字符串

.NET 6+ 引入了对查询字符串参数的内置支持。使用 [SupplyParameterFromQuery] 特性。

7. 布局 (Layouts)

布局组件用于为应用提供一致的 UI 结构，如页眉、页脚和导航菜单。

LayoutComponentBase

布局组件通常继承自 LayoutComponentBase。

@Body 属性

LayoutComponentBase 提供了一个 Body 属性 (RenderFragment 类型)，用于指定页面内容在布局中的渲染位置。

默认布局与指定布局

  • 默认布局在 App.razor (或 .NET 8+ 中是 Routes.razor) 中通过 <Router> 组件的 DefaultLayout 参数指定。
  • 可以在单个可路由组件上使用 @layout 指令指定不同的布局。

嵌套布局

布局本身也可以使用布局，形成嵌套布局。子布局的 @Body 会渲染到父布局的 @Body 中。

8. 依赖注入 (Dependency Injection - DI)

Blazor 内置了对依赖注入的支持，允许组件和服务请求它们所需的依赖项。

服务注册 (Program.cs)

服务在应用的 Program.cs 文件中注册。

在组件中注入服务 (@inject)

使用 @inject 指令将服务注入到组件中。

服务作用域

  • Singleton: 为应用程序的整个生命周期创建单个实例。
  • Scoped:
    • Blazor Server / Interactive Server: 作用域与客户端连接（SignalR Circuit）的生命周期相关。
    • Blazor WebAssembly / Interactive WASM/Auto: 作用域通常与用户的访问相关，类似于单例，但如果用户刷新页面，会创建新的作用域。在.NET 8 中，对于 WASM，”scoped” 服务的生命周期与根组件树相关联。
  • Transient: 每次请求时都会创建一个新实例。

9. JavaScript 互操作 (JS Interop)

允许 Blazor 应用调用 JavaScript 函数，也允许 JavaScript 调用 Blazor 应用中的 .NET 方法。

从 .NET 调用 JavaScript (IJSRuntime)

注入 IJSRuntime 服务，并使用其 InvokeAsync<TValue>(identifier, args) 方法。

JavaScript 函数必须在全局作用域 (window) 下可用，或者通过 JS 模块。

从 JavaScript 调用 .NET 方法 (DotNet.invokeMethodAsync)

.NET 方法必须是静态的，并且用 [JSInvokable] 特性标记。

MyBlazorAppAssemblyName 是你的 Blazor 项目的程序集名称。
也可以调用实例方法，但这需要将 .NET 对象引用传递给 JavaScript。

JSIsolatedRuntime (JS 模块隔离)

为了更好地组织 JavaScript 代码并避免全局命名空间污染，推荐使用 JavaScript 模块。通过 IJSObjectReference 来与模块交互。

10. 表单与验证 (Forms and Validation)

Blazor 提供了强大的表单处理和验证功能。

EditForm 组件

EditForm 组件用于处理数据编辑，并与验证集成。它需要一个模型 (Model 或 EditContext)。

内置输入组件 (InputText, InputNumber, 等)

Blazor 提供了多种与 EditForm 集成的输入组件：

  • InputText, InputTextArea, InputNumber, InputDate, InputCheckbox, InputSelect, InputRadioGroup。
    这些组件会自动处理验证状态并显示相应的 CSS 类。

数据注解验证

如上例所示，通过在模型属性上使用 System.ComponentModel.DataAnnotations 中的特性（如 [Required], [StringLength], [Range]）来实现验证。

ValidationSummary 与 ValidationMessage

  • ValidationSummary: 显示模型所有验证错误的列表。
  • ValidationMessage: 显示特定模型属性的验证错误。

自定义验证

  • 实现 IValidatableObject 接口的 Validate 方法。
  • 创建自定义 ValidationAttribute。
  • 在 EditForm 中使用 EditContext 直接操作验证消息。

11. 状态管理 (State Management)

管理应用程序或组件的状态是 Web 开发的关键部分。

组件内状态

最简单的状态管理，状态保存在组件的字段或属性中。适用于局部状态。

父子组件通信 (参数与事件回调)

  • 父组件通过参数将状态传递给子组件。
  • 子组件通过事件回调 (EventCallback) 通知父组件状态更改。

级联参数 (简单共享状态)

适用于在组件层级中向下传递不经常更改的共享状态（如主题信息、用户信息）。

Scoped DI 服务

创建一个 Scoped 服务，在其中存储和管理状态。不同组件可以注入此服务来访问和修改共享状态。这对于跨多个不直接相关的组件共享状态非常有用。
在 Blazor Server / Interactive Server 中，Scoped 服务生命周期与用户会话 (Circuit) 绑定。
在 Blazor WASM / Interactive WASM/Auto 中，Scoped 服务通常在用户访问期间表现得像单例，但如果刷新，则会重新创建。

浏览器存储 (LocalStorage/SessionStorage via JS Interop)

使用 JS Interop 与浏览器的 localStorage 或 sessionStorage 交互，以持久化状态。适用于需要在浏览器会话之间或即使用户关闭浏览器后仍保留的状态。

第三方库 (如 Fluxor, Blazor-State)

对于更复杂的状态管理需求，可以考虑使用专门的状态管理库。这些库通常实现 Redux 或类似模式，提供更结构化的方式来管理应用状态、操作和副作用。

12. 错误处理 (Error Handling)

标准 C# try-catch

可以在组件的方法中使用标准的 C# try-catch 块来处理预期的异常。

ErrorBoundary 组件

ErrorBoundary 组件用于捕获其子组件树中发生的未处理异常。

可以为 ErrorBoundary 定义自定义错误 UI。

13. 性能优化

避免不必要的渲染 (ShouldRender)

重写组件的 ShouldRender() 方法，根据逻辑判断是否需要重新渲染 UI。

组件虚拟化 (Virtualize 组件)

当需要显示大量数据时，Virtualize 组件可以显著提高性能。它只渲染当前可见的项目，并在用户滚动时按需加载和渲染其他项目。

可以提供 ItemsProvider 以支持异步按需加载数据。

延迟加载 (Lazy Loading for WASM)

对于 Blazor WebAssembly 应用，可以将不立即需要的程序集配置为延迟加载，以减少初始下载大小和加载时间。通常在 Program.cs 和项目文件中配置。路由可以配置为在导航到特定页面时才加载相关程序集。

预渲染 (Prerendering) 与 AOT (Ahead-of-Time Compilation for WASM)

  • 预渲染: 在 .NET 8 之前，Blazor Server 和 WebAssembly 应用都可以配置预渲染。服务器首先渲染组件的 HTML，快速显示给用户，然后 Blazor 应用接管。在 .NET 8 中，静态 SSR 和流式渲染提供了类似或更优的机制。
  • AOT 编译: 对于 Blazor WebAssembly，可以使用 AOT 编译将 .NET IL 代码直接编译成 WebAssembly 代码，而不是在运行时解释 IL。这可以显著提高运行时性能，但会增加应用下载大小和编译时间。

事件处理优化

  • 对于频繁触发的事件（如 onmousemove），避免在事件处理程序中执行复杂逻辑。
  • 如果不需要事件参数，可以使用 Lambda 表达式简化事件处理，避免不必要的 EventArgs 分配。
    @onclick="() => Count++"

14. Blazor 项目结构

一个典型的 Blazor Web 应用 (.NET 8+) 项目结构可能包含：

  • Program.cs: 应用入口点，配置服务和中间件。
  • App.razor: (.NET 8+) Blazor 的根组件，通常包含路由和全局布局/渲染模式配置。在旧版本中，App.razor 也承担此角色。
  • Components/ 文件夹:
    • Layout/: 包含布局组件，如 MainLayout.razor, NavMenu.razor。
    • Pages/: 包含可路由的页面组件。
    • Shared/: （可选，或直接在 Components/ 下）包含通用的、可重用的组件。
  • wwwroot/: 包含静态资源，如 CSS 文件、JavaScript 文件、图像。
    • app.css 或 css/app.css
    • bootstrap/ (如果使用)
  • Routes.razor: (.NET 8+ 新引入) 通常在这里定义 Blazor 的路由设置，取代了部分 App.razor 在旧版本中的职责。
  • appsettings.json: 配置文件。
  • .csproj 文件: 项目文件，定义项目依赖、目标框架等。

15. 使用 Dapper 操作数据库

Dapper 是一个简单、高性能的 .NET 对象关系映射 (ORM) 工具。它通过扩展 IDbConnection 接口，提供了一系列便捷的方法来执行 SQL 查询并将结果映射到强类型对象或动态对象。在 Blazor 应用中，Dapper 通常用于数据访问层，尤其是在需要对 SQL 有更多控制或追求极致性能的场景。

请注意关键点：

  • Blazor Server / Interactive Server / Static SSR / Streaming Rendering: 在这些服务器端执行的模式下，Blazor 应用可以直接访问数据库，Dapper 可以在服务器端代码（如服务类）中直接使用。
  • Blazor WebAssembly (WASM) / Interactive WebAssembly / Interactive Auto (WASM 阶段): Blazor WebAssembly 在客户端浏览器中运行，出于安全和技术原因，它不能直接访问数据库。在这种情况下，WASM 应用需要通过 HTTP 请求调用后端的 API (例如 ASP.NET Core Web API)，而这个后端 API 可以使用 Dapper 来与数据库交互。本节内容主要针对服务器端的数据访问场景。

15.1 Dapper 的优势

  • 性能卓越: Dapper 非常接近原始 ADO.NET 的性能，是目前最快的 ORM 之一。
  • 轻量级: 核心库非常小，对项目影响小。
  • SQL 控制: 开发者可以直接编写和优化 SQL 查询。
  • 易于使用: API 简洁，学习曲线平缓。
  • 扩展性: 可以轻松处理一对多、多对多等复杂映射。

15.4 Dapper 常用方法

  • QueryAsync<T>(sql, params, transaction, commandTimeout, commandType):
    执行查询并返回 IEnumerable<T> 类型的对象集合。T 可以是你的模型类。Dapper 会自动将查询结果的列名映射到 T 类型的属性名 (不区分大小写)。
  • QueryFirstOrDefaultAsync<T>(...):
    执行查询并返回结果集中的第一条记录，如果结果集为空则返回 default(T) (通常是 null)。
  • QuerySingleAsync<T>(...):
    执行查询并返回唯一的一条记录。如果结果集为空或包含多条记录，则抛出异常。
  • QueryMultipleAsync(...):
    用于执行返回多个结果集的 SQL 查询。
  • ExecuteAsync(sql, params, transaction, commandTimeout, commandType):
    执行不返回结果集的 SQL 命令 (如 INSERT, UPDATE, DELETE)。返回受影响的行数。
  • ExecuteScalarAsync<T>(...):
    执行查询并返回结果集第一行第一列的值，转换为类型 T。

参数化查询:
始终使用参数化查询来防止 SQL 注入攻击。Dapper 通过将匿名对象、动态对象或强类型对象作为 params 参数传递来实现这一点。

15.6 注意事项

  • 异步操作: 始终使用 Dapper 的异步方法 (QueryAsync, ExecuteAsync 等) 以避免阻塞线程，这对于保持 Blazor Server 应用的响应性至关重要。
  • 连接管理: using 语句确保数据库连接在使用完毕后正确关闭和释放。Dapper 通常会在执行命令时自动打开和关闭连接，除非连接已打开。对于事务，你需要手动管理打开和关闭。
  • SQL 注入: 始终使用参数化查询，永远不要直接将用户输入拼接到 SQL 字符串中。
  • 错误处理: 在服务层和组件层妥善处理数据库操作可能引发的异常。
  • Blazor WebAssembly: 再次强调，如果使用 Blazor WebAssembly，上述 Dapper 相关代码（服务类、连接工厂等）应该位于一个独立的 ASP.NET Core Web API 项目中。Blazor WASM 组件随后会通过 HTTP HttpClient 调用这些 API 端点。

16.7 替代方案：Entity Framework Core

虽然 Dapper 非常优秀，但对于需要更复杂的查询构建、变更跟踪、迁移管理等功能的场景，Entity Framework Core (EF Core) 可能是更合适的选择。EF Core 是一个功能更全面的 ORM。开发者也可以在同一个项目中根据需要混合使用 Dapper 和 EF Core，例如，使用 EF Core 进行写操作和复杂查询，使用 Dapper 进行高性能的读操作。

这个新增的章节应该能让你的 Blazor 报告更加详尽，特别是关于数据持久化方面的内容。

16. 总结与展望

Blazor 提供了一种强大且灵活的方式来使用 C# 和 .NET 构建现代 Web 应用程序。通过其组件模型、多种托管和渲染选项（尤其是在 .NET 8 中的增强）、以及与 .NET 生态系统的深度集成，Blazor 正在成为越来越多开发者的首选框架。
